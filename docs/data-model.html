Data Model & Entity Lifecycle

limn is an Electron app for agentic review of local git branches. Its data lives in two places at once — and that split is the whole architecture.

Generated from src/shared/types.ts · src/main/db/migrations.ts · src/main/ipc.ts

Two persistence philosophies

Nothing about the code under review is ever copied into the app. Diffs, file content, branch tips, commit logs — all read live from git on every load. Only the review layer built on top (sessions, comments, chat, approvals) is persisted, in a single local SQLite file.

read live · never persisted

Git is ground truth

Branch SHAs, the diff skeleton (DiffSkeleton, FileDiff, Hunk, DiffLine), commit logs and artifact file content are derived from git at request time. If the branch moves, the next load reflects it.

one SQLite file · WAL · single-writer

The review layer is stored

Sessions, comments, chat threads/messages, iterations, approvals, viewed/reviewed sets and prefs live in <userData>/limn.db via Node's built-in node:sqlite (DatabaseSync).

Keystone
Because git is authoritative, persisted rows store references and anchors (file paths, line content, SHAs) rather than copies. On every load the app re-anchors comments against the current diff and recomputes "what changed since you reviewed" — see approval baselines.

The identity invariant

A session is keyed by (repo, base identity, compare identity). Identity is computed per side: a branch follows its tip → b:<name>; a commit is frozen → c:<sha>.

function refIdentity(side) {
  return side.kind === 'branch' ? `b:${side.symbol}` : `c:${side.anchorSha}`;
}
Stay in lockstep
This rule is duplicated in JS (refIdentity) and in SQL (the base_ident / compare_ident generated columns). Multiple live sessions may share the same identity triple; findSession returns the newest live row as the resume hint. If the two identity definitions ever drift apart, session keying breaks silently.

Session lifecycle

Orchestrated in ipc.ts; transitions in db/sessions.ts. There is no hard delete in the normal flow — termination is a soft archive.

created →generate reviewed (n=1) →chat / fix iterated (n+1) →approve approved →start fresh archived

Creation — startSession

  1. Resolve both ref inputs to {kind, symbol, sha}; reject if base SHA == compare SHA.
  2. findSession on the identity triple (and archived_at IS NULL) — resume if found, else createSession. A commit side with an empty SHA throws (it would collapse all commits onto identity c:).
  3. A new session begins with no annotations, no iterations, no chats — just the ref pair and chosen AgentRef.

Generate — the initial review

  1. Acquire a per-repo lock (throws if the repo is busy).
  2. Read the live diff skeleton; run engine.generateReview(...), pumping EngineEvents to the renderer.
  3. mergeAnnotations reconciles the result against git — drops references to files no longer in the diff, guarantees 100% file coverage via a synthetic "Other changes" section.
  4. Persist annotations + discovered artifacts; set reviewedAtSha = head.
  5. resetIterations → write iteration n=1 (regenerate wipes stale n>1 rows so a later fix can't resume the wrong engine thread).
  6. reconcileChats — first generate creates the default 'review' + 'user' threads; later generates re-point the review chat at the newest iteration's engine session.

Retarget & terminate

retargetSession mutates a side (and thus the generated identity). archiveSession stamps archived_at; children persist but the row disappears from live lists and resume hints. "Start fresh" creates another live session for the same identity. Hard DELETE happens only on import-rollback (CASCADE drops children).

Comment status machine

A comment's status tracks its journey from reviewer note to resolved-by-agent. unresolvedCount counts queued + sent.

queued →sendBatch sent →agent run resolved
sent ⤺no resolution / error queued any anchored ⇄lineContent drift outdated
  1. New comment → queued.
  2. Send to agent: selected non-resolved comments are marked sent and persisted before the engine runs (a recoverable trail). The agent edits + commits on the branch itself.
  3. After the run, each sent comment is set resolved with a { verdict, note, commit }. A sent comment the agent didn't resolve resets to queued so it isn't lost; on error, all still-sent comments roll back to queued.
  4. outdated is set on load by reanchorComments when a stored anchor's lineContent no longer matches the diff; if it later re-matches, the comment is revived.

Approval baselines — "since you reviewed"

Three SHA markers let the UI highlight exactly what moved since the reviewer last looked, without storing any diff content.

MarkerSet whenDrives
reviewed_at_shaevery generate (= diff head)fallback baseline
approved_shaexplicit approvepreferred baseline
viewed_files.shafile marked viewedper-file "since" diff
artifact_approvalsartifact approvedper-artifact "since" diff

The effective baseline is approved_sha ?? reviewed_at_sha. When it differs from the current branch head, a "since" diff highlights the delta. A live-branch compare side also starts a watcher that updates the drift banner and notifies on background agent runs.
