Ref picker — 5 design options

Redesigning base / compare / worktree selection in the review header · local-review

The problem

1 · Inconsistent info. Base shows commits-in-branch; compare shows branches + worktrees. Different mental model per side.
2 · Inconsistent apply. Base applies automatically on click; compare applies on a button. You can't predict which.
3 · Overloaded button. The compare button means different things per state ("Check out", "Already checked out", "New review"…), and extra buttons appear.
4 · Worktree jumps. Picking a branch yanks the worktree selection to wherever that branch is checked out — motion you didn't ask for.

Design goals

  • One consistent picker for base and compare — same info (branches + commits), same interaction.
  • Viewing a diff never requires a checkout or a worktree choice. Worktrees only matter when you apply edits.
  • Changes apply on a button press, never automatically — the loaded review is expensive to rebuild and you may have a rich session open.
1Unified comparison dropdownRecommended
one trigger · both sides in one popover · GitHub-compare feel

A single base ← compare control. Opening it shows two ref slots up top; clicking a slot focuses it, and the same branch + commit list below drives whichever slot is active. One Update diff applies both. Worktree is a quiet footer — present only because edits need a target, never in the way of viewing.

How it solves the frictions

  • One control, one interaction — both sides use the identical branch+commit list.
  • Single Update diff button; both sides are staged, nothing auto-applies.
  • No state-dependent button labels — the action is always "update the diff".
  • Worktree demoted to a footer line; the diff loads regardless of checkout.
StrengthsMost compact. Mirrors GitHub's compare. Hardest to misread — there is exactly one way to change refs.
Watch-outsTwo-slot + list in one small popover needs careful layout. Switching the focused slot is a learned move.
2Two consistent triggers
base + compare stay separate buttons · identical picker behind each

Keep two header buttons so both sides are visible at a glance — but they open the exact same picker (filter, branches, commits, Apply). The asymmetry is gone: both stage and apply on a button, both show commits. Worktree is pulled out entirely into a small standalone status chip that you only touch to edit.

3Compare-primary, base secondary
compare is the subject · base is a quiet "vs main" reference

Acknowledge the asymmetry instead of fighting it: the compare branch is what you're reviewing, so make it the prominent chip. Base is a small vs main affordance next to it. Both open the same picker, but the visual weight tells you which one is the thing under review.

4Side-by-side comparison panel
one wide panel · base column + compare column · live diff-stat preview

Clicking the comparison opens a wider two-column sheet: base on the left, compare on the right, each with its own branch + commit list. A live diff-stat strip previews what you'd get before you commit to it, and one Update diff applies. Best when you actually want to explore commits on each side side-by-side.

Watch-outsLargest surface — heavier than a quick branch switch deserves. The live preview needs a cheap diff-stat call on every hover/select.
5Search-first command palette
type to find any ref · unified results · assign to a side

The header shows main ← feat/branch as editable chips; clicking one (or a hotkey) opens a single search box. You type once and it fuzzy-matches across branches, commits (SHA / subject), tags, and HEAD~N — results are unified, each row says what it is and which side it'll fill. Pick, then Apply. Worktrees never appear here at all; they surface once, contextually, the first time you apply an edit.
