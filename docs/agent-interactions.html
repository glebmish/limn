Agentic Interactions

How Limn constructs prompts, propagates context, and exchanges structured inputs & outputs with an AI agent — Claude (Agent SDK) or Codex (Codex SDK).

Source: src/main/engines/ · orchestrated by src/main/ipc.ts · shared types in src/shared/

00Overview

Every interaction with an AI agent flows through one narrow interface. The renderer never talks to an SDK; it sends an IPC request, the main process selects an engine, and the engine drives the underlying CLI/SDK. Two SDKs sit behind one abstraction:

Renderer
IPC call
generate / sendChat / sendBatch + an opId
Main · ipc.ts
Orchestrate

engines/
ReviewEngine
prompt + schema + tools → SDK
SDK / CLI
Agent
claude / codex subprocess

The agent never runs over a raw HTTP call from this codebase — it runs as a local subprocess (the claude / codex CLI) the SDK manages. So "the wire" here is the prompt string + JSON Schema + tool definitions going in, and a normalized event stream + a schema-validated payload coming out. That contract is what this document explains.

01Git is ground truth

The single most important invariant. The diff the reviewer sees is always computed from git diff — never from the agent. The agent only annotates (sections, narration, risk flags, diagrams, spec cross-check) and, on write-enabled turns, edits and commits on the branch — after which the app re-reads git.

Consequence for I/O design: the agent's structured output is reconciled against the real diff (validate.ts → mergeAnnotations). Any reference to a file not in the diff is dropped; every file in the diff is guaranteed a section. The agent can describe the change but can never assert what changed or alter what you see.

02The EngineRun envelope

Every operation returns the same dual-channel envelope. src/main/engines/types.ts

interface EngineRun<T> {
  events: AsyncIterable<EngineEvent>                      // live progress stream
  result: Promise<{ value: T; sessionId: string }>     // terminal validated payload
  cancel: () => void                                     // AbortController.abort()
}

events  push → pull

Live EngineEvents: status, tool-call lifecycle, streamed text, agent actions, approval requests. Pumped to the renderer over IPC channel op:event as they arrive.

result  awaited

Resolves once with the validated terminal payload + the engine sessionId. Rejects on any failure (including an invalid schema parse).

The two channels run concurrently: ipc.ts calls pumpEvents() to drain events while it awaits result. The bridge between push-style SDK callbacks and the pull-style AsyncIterable is EventQueue:

// EventQueue — single-consumer FIFO. push() either hands the event to a
// waiting next(), or buffers it. close() drains waiters with {done:true}.
push(ev) { if (this.closed) return; const w = this.waiters.shift(); w ? w({value: ev, done: false}) : this.buffer.push(ev) }
Errors do not reject the iterable. An error travels as both an {type:'error'} event and a rejected result promise. Events pushed after close() are dropped.

04Prompt construction

Prompts are plain-string builders — no templating engine. engines/prompts.ts. Four builders cover every turn; each one's job is to frame the task, supply git-derived facts, and state the output contract.

buildReviewPrompt(req) — the initial review

Frames the agent as a review guide, instructs an explore-first pass, injects a precomputed changed-files list (status, ±counts, hunk ranges, merge-base/head SHAs) and an artifact block, then states a 7-point output contract.

`You are the review guide for a local branch. ... help a human review
the branch \`${req.branch}\` against \`${req.base}\` ...

EXPLORE FIRST — you have full read access to the repo:
- Read the changed files in full, not just the diff hunks listed below.
- Find callers/usages of changed functions (grep), related tests, and read them.
- Use \`git log ${req.base}..${req.branch} --oneline\` and \`git show\` ...
${artifactBlock}
Changed files (N) between merge-base abc1234 and def5678:
- src/foo.ts (modified, +12/−3) hunks: @@ 10,6 @@, @@ 40,8 @@
 ...
Then produce the structured review:
1. Group ALL changed files into logical sections ... each file in exactly one section ...
2. For each section write "desc" and "what" (plain-language narration) ...
3. Flag risky hunks in "flags" (risk:true = correctness/security; false = worth a look) ...
4. Optionally a "diagram": 2-5 nodes [label, kind, sub] ...
5. "title" + "summary" ...   6. "planMap" if a spec/plan exists ...   7. "questions" ...`

Note the prompt restates the schema's intent in prose and the schema is enforced separately (§06) — the prose teaches the model how to fill the fields well; the schema guarantees the shape.

buildChatPrompt(message, anchor?) — read-only Q&A

A thin conversational wrapper. If the reviewer clicked a specific spot, describeAnchor() renders it into prose context.

`${message}${anchor ? `\n\nContext — the user is asking about ${describeAnchor(anchor)}.` : ''}

Answer conversationally and concisely ... You may read files and run read-only
git commands ... Do NOT modify any files.`

describeAnchor turns each of the 7 CommentAnchor kinds into a sentence the agent can act on — e.g. diff → "src/foo.ts line 42 (new side, hunk @@..@@): `const x = …`", section → "review section \"sec-2\"".

buildSeededChatPrompt(ctx, message, anchor?) — fresh chat, no session to resume

Used when a chat thread targets a different agent than the one that produced the review, so there's no engine session to resume (see §07). It re-orients the new agent from scratch — branch/base, the review summary so far, read access — instead of relying on resumed conversation state.

buildBatchPrompt(comments, steer?, context?) — apply queued comments

Lists each queued comment with its stable id, anchor, text, and reply thread, plus an optional reviewer steer. Crucially, it instructs the agent to act via tools, not to return JSON:

`The reviewer is sending you N comment(s) to handle on the current branch.
Comments:
1. [id: c-17] on src/foo.ts:42:
   "this off-by-one looks wrong"
   thread: user: are you sure? | agent: checking…

Handle them using your tools:
- Address each comment by editing the code, or answer with reply_to_comment ...
- When you edit code, commit it with commit_changes — pass a message and the
  resolutions (each: commentId, verdict, note). Verdicts: addressed/reworked/skipped.
- For comments handled without a code change, call resolve_comment ...
- Finish with a 2-3 sentence summary of what you did.`
The comment id is the join key that survives the whole round-trip: prompt → agent's resolve_comment/commit_changes call → the DB row's resolution. The model is told the ids explicitly so its tool calls can reference them.

05Context propagation

"What does the agent know?" is assembled from several sources and threaded through ReviewRequest / ChatTurn. There are four distinct channels:

ChannelCarriesHow it reaches the agent
Prompt bodyBranch/base, changed-files list, artifact paths, comment list, anchor prose, review summaryInlined as text by the prompt builders
Engine sessionThe agent's own prior conversation (its exploration, its reasoning)resume (Claude) / resumeThread (Codex) — §07
Working directoryThe repo itself — full read access via Read/Grep/Glob/Bashcwd / workingDirectory: repo
Live tool readsDB state at call time (current comments, the review, iterations)Limn tools (list_comments, get_review) query the DB on demand

This is why the prompt can stay lean: it doesn't dump the whole repo or all prior comments. It names the branch and the changed files and lets the agent pull what it needs through tools and the filesystem. The ChatTurn struct is the carrier:

interface ChatTurn {
  repo: string
  engineSessionId?: string     // resume this session, else start fresh + seed
  model?: string; reasoningEffort?: ReasoningEffort
  message: string; anchor?: CommentAnchor; context?: ChatContext
  tools?: AgentToolHost         // the limn tool layer for this turn
  writeEnabled?: boolean        // branch + clean tree preconditions met
  opId?: string                 // keys the approval registry
  executionMode?: ExecutionMode  // the chat's autonomy tier
}
opId is the correlation key that ties an in-flight operation together across processes: it cancels the run (activeOps), routes approval decisions back to the parked promise (awaitDecision), and tags every op:event so the renderer attributes events to the right turn.

06Sessions & resume

Each agent SDK keeps its own conversation store keyed by a session/thread id. Limn captures that id and reuses it so chat and batch continue the same conversation that produced the review — the agent still "remembers" the files it read and the reasoning it did.

The branching logic is identical in both engines:

const prompt = write
  ? turn.message                                // batch carries its own fully-built prompt
  : turn.engineSessionId
    ? buildChatPrompt(turn.message, turn.anchor)            // resume → thin wrapper
    : buildSeededChatPrompt(turn.context ?? …, turn.message, turn.anchor)  // fresh → re-orient
// …
const thread = turn.engineSessionId ? codex.resumeThread(id, opts) : codex.startThread(opts)
No fallback if the session is gone. chat/batch resume the review's session id from the last iteration. If the CLI store pruned it, or the engine was switched mid-review, resume targets a missing/wrong session — there is no automatic re-seed.

07Structured I/O — wire schema vs. internal types

The review output is the one place the agent must return a strict structured payload. There are two type worlds, deliberately kept separate. engines/schema.ts · shared/types.ts

Wire schema Zod → JSON Schema

What the SDK is asked to return. Shaped around two validator constraints (below). Handed to the SDK via z.toJSONSchema().

Internal types shared/types.ts

What the app stores & renders. parseReviewOutput() Zod-parses the wire payload and translates wire → internal.

The two SDKs' structured-output validators impose constraints that the internal types violate, so the wire schema bends to them and a post-parse step un-bends:

ConstraintWire schema doesPost-parse fixes
Claude CLI silently drops schemas with prefixItems → no tuplesDiagram nodes travel as {label, kind, sub} objectsmapped to [label, kind, sub] tuples (DiagramNode)
Codex OpenAI strict mode requires every prop in required, forbids recordsoptionals are .nullable() (not .optional()); per-file notes are an array of {file, note}null → undefined; array → Record<file, note>
export function parseReviewOutput(raw: unknown): ReviewAnnotations {
  const wire = reviewWireSchema.parse(raw)          // throws on shape mismatch — NO retry
  return {
    title: wire.title, summary: wire.summary,
    sections: wire.sections.map((s) => ({
      …s,
      diagram: s.diagram?.map((n) => [n.label, n.kind, n.sub]) ?? undefined,   // object → tuple
      plainNotes: s.plainNotes ? Object.fromEntries(s.plainNotes.map(p => [p.file, p.note])) : undefined,  // array → record
      flags: s.flags.map(f => ({ …f, hunkRange: f.hunkRange ?? undefined }))     // null → undefined
    })),
    planMap: wire.planMap ?? undefined,
    questions: wire.questions.map(q => ({ …q, context: q.context ?? undefined })),
  }
}
No retries on a bad parse. An invalid payload throws → the engine pushes an error event → result rejects → the IPC handler reports failure. One bad parse fails the whole op.

Reconciliation against git — the final validation

This runs in ipc.generate, not in the engine. mergeAnnotations(skeleton, parsed) enforces "git is ground truth": drops files/flags referencing files not in the diff; enforces one section per file; drops empty sections; sweeps any unassigned diff file into a synthetic "Other changes" section (100% coverage); sanitizes planMap cross-references. Warnings surface to the UI as status events.

08The tool layer — one set, hosted two ways

Beyond reading files, the agent acts on the review through engine-agnostic tools defined once and reflected into each SDK. engines/tools.ts

// and returns the text the model sees.

The same LIMN_TOOLS array reaches each SDK differently:

Claude in-process MCP

createSdkMcpServer + tool(name, desc, shape, handler) consumes the Zod shape directly. Allowed names are prefixed mcp__limn__*.

Codex app-server + MCP

A per-turn localhost MCP server (registerCodexTurn) reflects the same shapes into JSON Schema; codex app-server is pointed at it via config.mcp_servers.limn.url.

Write tools are withheld unless the turn is write-enabled — the agent silently degrades to review/comment-only rather than failing:

export function limnAllowedToolNames(writeEnabled = false): string[] {
  return TOOL_IMPLS.filter(t => writeEnabled || !t.write).map(t => `mcp__limn__${t.name}`)
}

Each tool call validates its args against the Zod shape, runs the handler, and — if the handler produced one — emits a live AgentAction event and collects it for persistence on the chat message (so chips rebuild on reload):

async call(name, args) {
  const parsed = z.object(impl.input).safeParse(args)         // validate at the boundary
  if (!parsed.success) return { result: `Invalid arguments…`, isError: true }
  const out = await impl.run(ctx, parsed.data)
  if (out.action) { ctx.emit({ type: 'action', action: out.action }); collected.push(out.action) }
  return out
}

When a tier requires a go-ahead, the engine parks on an approval and emits a request; the reviewer's answer (a blocking card in chat) routes back through IPC to unblock the parked promise. engines/approvals.ts

export function awaitDecision(opId, request, emit): Promise<ApprovalDecision> {
  emit({ type: 'approval_request', request })          // → renderer shows a blocking card
  return new Promise(resolve => parked.set(key(opId, request.id), { opId, resolve }))
}
// respondApproval IPC → resolveDecision(opId, requestId, decision) → unblocks the promise
// clearPending(opId) auto-denies every parked request on cancel / turn end (no leak)

On Claude, the SDK's canUseTool callback is the gate: limn + read-safe tools auto-allow; Bash/Edit/Write park on awaitDecision. The raw tool call is normalized into an engine-agnostic ApprovalRequest (toApprovalRequest) so the same card renders for either engine.

No timeout anywhere. A parked approval — and a wedged run — waits until the reviewer answers or cancels.

10Event normalization

Each SDK has its own native event stream. Every engine maps it into one 7-variant union so the renderer is engine-blind. shared/types.ts

type EngineEvent =
  | { type: 'status'; text }              // "Codex is working…", reasoning snippets, notes
  | { type: 'tool'; call: ToolCall }       // tool-call lifecycle (run → ok/err), same id
  | { type: 'text'; text }                // streamed assistant text (chat)
  | { type: 'action'; action: AgentAction } // Limn tool effect (focus, comment_added, code_committed…)
  | { type: 'approval_request'; request }  // needs reviewer go-ahead
  | { type: 'done' }
  | { type: 'error'; message }

claude.ts · toEvents(msg)

One SDKMessage → 0+ events. An assistant msg's content blocks split into tool_use (→ run) and text. A later user msg's tool_result settles the matching call by id (→ ok/err). system/init → a status.

codex.ts · toEvent(ev)

One ThreadEvent → 0/1 event. mcp_tool_call / command_execution / file_change map to the tool lifecycle; reasoning → status; agent_message → text; turn.failed/error → error.

Both feed the same downstream: the tool-call lifecycle is emitted twice per call — a run on start, an ok/err on completion, sharing one id — and large tool outputs are clamped (clampOut, ~30 lines / 2000 chars) with a "show N more" remainder.

Because each call emits two events, both the live renderer and the persistence path fold the event stream into settled rows with one pure function — same input, same output — so live and reloaded views agree. shared/toolcalls.ts

The display verb (read/grep/edit/bash/list/other) is derived from the raw tool name via a regex table — so commit_changes reads as bash, list_comments as list, the MCP prefix is stripped first, and unknown names fall through to other.

12Flow — Generate (initial review)

ipc.generate

  1. Acquire the per-repo lock (repoLocks); throw if another op is running for the repo.
  2. Build the diff skeleton (getDiff), load review state + artifacts.
  3. engine.generateReview({ repo, branch, base, diff, artifacts, model, effort }).
  4. pumpEvents(opId, run.events) streams status/tool events to the renderer while await run.result.
  5. mergeAnnotations(skeleton, value) → validated annotations + warnings (emitted as status events).
  6. Persist discovered artifacts; updateSessionMeta (annotations/title/summary, reviewed_at_sha = head); reset to iteration 1 recording the engine session id.
  7. Emit op:result {kind:'review', reload:true}; desktop-notify if unfocused. Finally: release lock, clear parked approvals.

13Flow — Chat

ipc.sendChat

  1. Lock the repo. Build a createToolHost bound to {db, sessionId, threadId, opId, repo, agent, writeEnabled:false, emit}.
  2. engine.chat({ engineSessionId: thread.engineSessionId, message, anchor, tools, opId, executionMode, context }) — resume if the thread has a session, else seed from context.
  3. Stream text + tool + action + approval_request events; await the final text.
  4. Persist the agent message with its collected actions & settled tool calls. Emit op:result {kind:'chat'}.

14Flow — Batch (apply comments)

ipc.sendBatch — chat with write tools

  1. Lock the repo. Compute writeEnabled from preconditions: compare side is a branch, tree is clean, repo is on that branch. If unmet, the agent runs write-disabled (review/comment-only) — it doesn't fail.
  2. Mark the selected comments sent and persist (a crash leaves a recoverable trail).
  3. Build the tool host (now writeEnabled, carrying engineSessionId for the commit's iteration) and buildBatchPrompt(comments, steer, context?).
  4. engine.chat({ …, tools, writeEnabled, executionMode }) — the agent edits, calls commit_changes (commits on the branch + records an iteration), and resolve_comment/reply_to_comment per comment id. The app never commits — the agent does.
  5. Reconcile: any comment still sent (un-addressed) rolls back to queued so it isn't lost.
  6. Emit op:result {kind:'chat', reload:true}; the next loadSession re-diffs against git and tags new commits as "since you reviewed".
The loop closes on git, not the agent. The agent reports what it did via tool calls, but what the reviewer then sees as "changed" is re-derived from git diff — the agent's account of its own work is never trusted as the source of truth.

16Invariants & sharp edges

  • Concurrency: one agent op per repo (repoLocks). The watch-mode poller skips a locked repo so the app's own commits don't trigger a spurious reload. EventQueue is single-consumer.
  • Cancellation: cancel() → AbortController.abort() (Claude abortController, Codex signal). No timeout — a wedged run hangs until cancelled.
  • Errors: propagate as both an error event and a rejected result. No retry/backoff anywhere, including on invalid structured output — one bad parse fails the op.
  • Session resumption: chat/batch resume the review's session from the last iteration. If it's gone or the engine was switched, resume targets a missing/wrong session with no fallback.
  • Codex output fragility: the review payload rides the final message text; only the ```json fence-stripper stands between a fenced response and a hard failure.
  • Validation runs in IPC, not the engine: engines return raw payloads; ipc.generate reconciles them against git. Engines are deliberately thin.
