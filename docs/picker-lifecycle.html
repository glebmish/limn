The reframe

The agent does its analysis on real files in a working directory — so checkout gates the entire AI value of the app, not just edits. That makes checkout the natural "I'm committing to work on this branch" moment, which is exactly the right place to draw the ephemeral → persistent line.

Free — no checkout, no session

  • View the base ← compare diff
  • Read sections, files, commit list
  • (maybe) jot manual comments

Gated — needs the compare branch checked out

  • Generate guided review (agent reads real files)
  • Chat about the code
  • Apply queued comments (edit & commit)

A What a pointer change does — no apply button

You're right: with a read-only diff there's nothing destructive to confirm, so picking a ref applies immediately. Viewing is cheap and safe (it never touches git or a session), so a staging button would only add a click. Here's exactly what happens the moment you pick a new base or compare ref.

B The lifecycle spine — ephemeral until checkout

One clean rule: checkout is the moment a draft becomes a real session. It's also the moment the AI unlocks — same gate, one mental model. Crucially, the session is bound to the (base, compare) pair, not to the worktree: you can later move the branch to a different worktree and the session is untouched.

Browse
Ephemeral preview
Pick refs, read the diff, skim commits. No session row, no checkout. Disposable.
Check out ⟶ persists
Real session
Check the compare branch into a worktree. The draft mints a session and the agent unlocks — one action, both effects.
Work
Generate · chat · apply
Agent reviews real files, you queue comments, it edits & commits. Session accrues iterations.
Sessions ≠ worktrees. The worktree only answers "where is this branch on disk right now." Move it elsewhere later and the session — its comments, history, approvals — rides along with the pair, not the folder.

C Where the worktree lives — three UI models

All three keep session and worktree visually distinct (they're different things) but acknowledge they're both "working context." They differ in how much chrome sits in the header vs. how much surfaces contextually where you act. Recommended first.

RecommendedC1 · Two adjacent chips: Session · Edit-targeton the header right; the worktree popover has no branch row

Right cluster = working context: a Session chip and an Edit-target chip, side by side but separate. In a draft, Session reads Draft and Edit-target reads read-only ▾. Opening Edit-target gives a pure worktree picker (clean worktrees + New) — checking out persists the session (Draft → #6) and unlocks the AI. You can also trigger the same checkout from the Generate CTA.
