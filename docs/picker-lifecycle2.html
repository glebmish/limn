Lifecycle, refined — comments save before checkout

Saving and the agent are two separate gates · limn

Got it — two independent gates, not one line. Leaving a comment saves it (you never lose annotation work), and that's separate from checkout. Checkout isn't what makes your comments persist — it's only what lets you relay them to the agent (and generate / chat). So you can fully annotate a read-only diff, come back to it later, and only check out when you're ready to put the agent to work.

One axis is save (do your comments persist?) — tripped by your first comment. The other is agent (can the AI act?) — tripped by checkout. They're orthogonal, which gives three meaningful states.

3 The one decision this creates

If comments save before checkout, the Annotated draft needs a home. Does a comment-only review show up as a session (in the sessions list / hub), or is it a lighter draft that only becomes a listed session once you check out?

RecommendedD1 · First comment mints a session, badged "Draft"one storage tier; matches the existing transient-review plan

Commenting materializes a real session row immediately, shown everywhere with a Draft · read-only badge until checkout clears it. The hub lists it (so a half-done annotation isn't invisible), just visibly distinct from checked-out sessions. This is the path the current transient-review spec already lays out — persist on first write — with checkout layered on as the agent gate.

StrengthsComments always live on one thing (a session) — no second storage path, no promotion step. Drafts are findable in the hub. Smallest delta from the planned architecture.
Watch-outThe sessions list can collect abandoned comment-only drafts. Mitigate with the Draft badge + easy archive (already supported).
AlternativeD2 · Comments save as a draft; checkout promotes to a sessiontwo tiers; keeps the session list "real work only"

A comment-only review is stored as a draft keyed to the (base, compare) pair but not listed as a session. Checkout promotes it into a full session that appears in the hub. Honors your literal "no session" — the list only ever shows checked-out, agent-capable work.

StrengthsCleanest sessions list — only real, checked-out work appears. Matches the "no session until checkout" phrasing exactly.
Watch-outA second persistence tier + a promotion step (more code, more edge cases: what if two drafts exist for a pair? where do draft comments surface?). Drafts are easy to forget if they're not in the list.

So: do you want D1 (comment → a Draft-badged session, recommended) or D2 (comment → an unlisted draft, promoted on checkout)? Once that's settled I think we have the full model and I can write the spec.
